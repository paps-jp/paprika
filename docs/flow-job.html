Job Lifecycle

POST /jobs 到着 → 直接配信 or キュー → 再投入 → worker 実行 → 結果永続化と課題分類。mode は fetch / codegen-loop / rerun の 3 系統。

概要
  • POST 時に空き lane があれば直接配信(PATH A)。無ければ row は queued のままになり、各 Hub の 再投入ループが 3 秒ごとに拾う(PATH B)。
  • WorkerJobComplete を受けたら、占有率ベースの課題分類(全画面オーバーレイで content が壁に塞がれていないか)を経て completed / review に振り分け。
  • 二重配信は 3 重ガード(原子 CAS / worker_id IS NULL 確認 / age-grace)で防止。ACK が来ない場合は 15 秒で queued に戻して再投入に委ねる。

codegen-loop / rerun はフリート capacity チェックを経由しない(GPU gate と別経路)。fetch だけが満杯時に 503(fleet at capacity)を返す。

sequenceDiagram participant L as _redrive_loop
(各 Hub, 3 秒間隔) participant S as Store participant H as Hub participant W as Worker Note over L: POST 時に空き無し → row は queued のまま loop 3 秒ごと L->>S: list queued (age >= 2s で claim 競合回避) L->>L: oldest-first sort (FIFO) L->>H: pick_worker() alt 空きあり L->>S: claim_queued_job CAS L->>W: HubAssignJob (POST と同じ mirror) W-->>H: WorkerJobAccepted else 空きなし L->>L: skip → 次ループ end end

server/hub/_redrive.py:282–463

なぜ再投入が必要か: POST が fetch を inline で配信する間にクライアント切断や cross-hub 並列分散で中断された queued ジョブが死蔵される事故が頻発(失敗の大半が「180 秒以上 queued のまま割り当てなし」)。再投入は 3 重ガード(CAS / worker_id IS NULL / 90s age-grace)で二重配信を防ぐ。

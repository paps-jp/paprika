Asset / Storage

MinIO がアセット SoT。Hub のローカル /data/jobs は 180 秒で evict される bounded cache。DELETE は MinIO 連動削除。MariaDB 行は ephemeral(クライアントが消す)、MinIO は durable。

概要
  • アセット SoT は MinIO。Hub のディスクは bounded cache で、180 秒経過 + MinIO 側の存在確認後に evict。 どの Hub からでも同じものが見える。
  • upload 時に .meta sidecar も MinIO にミラー。これが無いと evict 後に source_url などのメタデータを失う(disk-full 事故からの修正)。
  • DELETE は MinIO 連動削除: 行を消す前に MinIO の jobs/{id}/ prefix を一括削除。 過去 leak していた orphan を根絶。

2. Worker → Hub → MinIO upload

sequenceDiagram participant W as Worker participant H as Hub (routes/jobs/artifacts.py) participant FS as /data/jobs (cache) participant M as MinIO W->>H: POST /jobs/{id}/assets/{name} (binary + meta hdrs) H->>FS: write asset H->>FS: write .meta sidecar (content-type, source_url, original-name, mtime) par 非同期 mirror H->>M: PUT jobs/{id}/{name} H->>M: PUT jobs/{id}/{name}.meta end H-->>W: 201 Created
.meta sidecar も mirror が必要: 過去の disk-full 事故以降、保存時に asset と一緒に .meta も MinIO へ。これが無いと evict で source_url が消える。

事故の系譜: cache が動画で 22 GB/hub まで育って 3 Hub 全停止。修正=(1) sidecar も同時 mirror、(2) prefix_exists 確認後だけ削除、(3) ホスト cron フォールバック。get_storage_dir() は素の docker exec では相対パスを返す罠あり(config.data_dir=/data/jobs を set してから)。
